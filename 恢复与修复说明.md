# 恢复与修复说明

## 场景一：镜像文件损坏

**症状**：`verify` 报告某个镜像文件"内容不一致"。

**处理**：
```
python archive_keeper.py repair D:\我的归档 --dry-run   # 先看修复计划
python archive_keeper.py repair D:\我的归档              # 确认后执行
```

系统会用主副本覆盖损坏的镜像文件，修复后自动复检。

## 场景二：镜像文件丢失

**症状**：`verify` 报告某个镜像文件"文件缺失"。

**处理**：同上，`repair` 会从主副本重新复制。

## 场景三：主副本文件损坏

**症状**：`verify` 报告主副本与基准不符。

**处理**：
1. 如果镜像和快照中有正确版本，`repair` 会自动用多数一致的版本修复主副本。
2. 如果不确定，先查看快照中的历史版本：
   ```
   dir D:\我的归档\snapshots\
   ```
   找到最近的快照目录，手动对比内容。

## 场景四：无法自动判定（冲突）

**症状**：`repair` 提示"无法自动判定，需要人工介入"。

**原因**：多个副本内容不同，系统无法通过投票确定正确版本。

**处理**：
1. 查看报告中列出的各源哈希值。
2. 手动打开各副本中的文件进行对比。
3. 确定正确版本后，手动复制到 master。
4. 重新运行 `sync` 同步到镜像。

## 场景五：误删文件后恢复

如果你不小心删了 master 中的文件：

1. **从镜像恢复**：如果还没运行过 sync，镜像中还有。
   ```
   copy D:\我的归档\mirrors\mirror_1\路径\文件名 D:\我的归档\master\路径\
   ```

2. **从快照恢复**：快照是冻结的历史版本。
   ```
   dir D:\我的归档\snapshots\          # 查看可用快照
   copy D:\我的归档\snapshots\20260115_143000\路径\文件名 D:\我的归档\master\路径\
   ```
   注意：快照文件是只读的，复制后目标文件是可写的。

## 场景六：整个归档目录混乱

最后手段：
1. 找到最新的完好快照。
2. 清空 master 和 mirrors。
3. 把快照内容复制回 master。
4. 重新运行 `maintain`。

## 原则

- 永远先 `--dry-run`，再实际操作。
- 不确定时，先看快照里有没有你需要的版本。
- `repair` 不能判定的，不要强行修，手动确认后再操作。
- 每次修复后 `verify` 确认结果。
