# 设计说明

## 选型理由

### 纯 Python stdlib

- 目标环境为 Windows 11 + Python 3.9+，无需安装第三方包。
- 所有功能（哈希、文件操作、JSON、日志）均来自标准库。
- 如果未来需要迁移到 Linux/macOS，代码无需修改。

### SHA-256 作为哈希算法

- 对 <10GB 数据量，性能不是瓶颈（约 200MB/s）。
- 碰撞概率可忽略。
- 如需更快可在 config.json 中换成 blake2b（已内置支持），只需修改 `hash_algorithm` 配置。

### 单文件设计

- 一个 .py 文件包含全部逻辑，拷贝即用，不需要 pip install。
- 降低"工具本身出问题"的风险。

## 目录结构设计

```
archive_root/
├── master/           # 主副本：用户唯一写入口
├── mirrors/
│   ├── mirror_1/     # 镜像副本 1：系统维护，用户不应修改
│   └── mirror_2/     # 镜像副本 2：系统维护，用户不应修改
├── snapshots/
│   └── 20260115_143000/  # 时间戳命名，冻结后只读
│       └── _snapshot_baseline.json  # 快照自带校验基准
└── .archive/
    ├── config.json       # 集中配置
    ├── baseline.json     # 主基准（最近一次完整同步后的校验和）
    ├── logs/             # 机器日志（每次操作一个文件）
    └── reports/          # 人类报告（每次操作一个文件）
```

**为什么固定两份镜像？**
固定两份镜像可以让日常同步和恢复路径更直接，也减少用户把镜像数量当成运行时开关带来的复杂性。

**为什么快照有自己的 baseline？**
快照是独立的可信锚点。如果主基准被污染，快照基准仍然可用。

## 状态模型

每个文件在系统中有以下可能状态：

| 状态 | 含义 | 来源 |
|------|------|------|
| 完全一致 | 所有副本 + 基准哈希相同 | verify |
| 内容不一致 | 某副本哈希与基准不同 | verify |
| 文件缺失 | 基准中有记录但副本中找不到 | verify |
| 未纳入基准 | 文件存在但基准中无记录 | verify |
| 多余文件 | 镜像中有但主副本和基准中都没有 | verify |
| 基准过时 | 镜像与主副本一致但与基准不同 | verify |

## 修复判定策略

**分层判定**：`repair` 不再把主副本和镜像当成独立票源，而是把它们视为同一条“当前链”；基准和最新快照作为历史证据。

**自动修复条件**：
- 当前链现存副本一致，只是镜像缺失或不可读 → 自动补齐镜像
- 某个现存版本得到基准或快照支持，且至少有一个当前副本还保留这个版本 → 自动修复其他偏离副本
- 显式运行 `repair` 时，主副本和基准都不存在但镜像仍有旧文件 → 清理镜像遗留

**停止条件**：
- 当前链整体一致，但与历史证据冲突
- 历史证据分别支持不同的当前版本
- 没有足够的现存副本支撑安全恢复

**原则**：宁可不修也不乱修。

## 同步策略

- 以 master 为唯一真相源。
- 只有两份镜像都复制并校验成功后才原子更新基准；部分失败时保留旧基准。
- 删除保护：master 中删了的文件，默认不从镜像中删（防误删扩散）。
- 每次拷贝后立即校验，确保写入正确。

## 统一维护安全门

- `maintain` 在任何快照、同步或 baseline 更新前，先核对 baseline 已跟踪的主副本文件。
- baseline 缺失、schema 不完整或哈希算法不匹配时直接停止；只有显式 `sync` 可以在确认 master 正确后重建基准。
- 已跟踪文件缺失、不可读或哈希变化时立即停止，并保留镜像、快照和 baseline。
- 新路径可以由 `maintain` 自动纳入；确认过的已有文件修改通过显式 `sync` 接纳。
- 完整性校验仅为警告时不自动调用清理；镜像残留必须通过显式 `repair` 确认。

## 快照策略

- 安全门通过且主副本非空时，maintain 自动创建快照。
- 文件先复制到隐藏暂存目录并逐项校验；清单落盘后通过同目录原子重命名发布。
- 主副本目录枚举错误会直接终止快照，不能被当成空目录或完整扫描。
- 复制、哈希验证、清单写入或发布任一步失败，都删除暂存结果且不执行旧快照淘汰。
- 快照文件设为只读（流程层面的保护，非加密锁定）。
- 完整快照超过上限（默认 10 个）后自动清理最旧的。
- 快照自带独立基准，不依赖主基准。

## 日志策略

- 双通道：机器日志（DEBUG 级别，含完整路径和哈希）+ 人类报告（摘要）。
- 每次操作一个日志文件，文件名含时间戳和命令名。
- 报告直接输出到终端，同时保存文件。

## 幂等性

- sync：已一致的文件不重复拷贝。
- snapshot：同秒内不会覆盖已有快照（自动追加唯一序号）。
- verify：纯读操作，无副作用。
- repair：已修复的文件不重复操作。
- maintain：重复运行结果稳定。

## 外部依赖

**无。** 全部基于 Python 3.9+ stdlib。

如果未来需要：
- 更快的哈希 → 可换 blake2b（已在 hashlib 中）
- 跨设备同步 → 可在 sync 层包装 rsync/robocopy
- 加密保护 → 可在快照层包装 7z/gpg
- 图形界面 → 可包装 tkinter（stdlib 自带）

这些扩展不需要改动核心逻辑，只需在接口层添加。
