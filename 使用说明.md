# Vault 使用说明

## 这是什么

Vault 是一个本地归档维护工具。它帮你在同一块硬盘内保留多份副本，定期检查文件完整性，发现损坏后尽可能自动修复。

适合场景：重要但不常变动的文件（合同、照片、财务文档等），数据量在 10GB 以内，每隔几个月手动维护一次。

## 推荐使用方式

日常使用请双击仓库里的：

```
点我开始.cmd
```

窗口打开后按按钮操作：

1. 选择一个归档文件夹
2. 初始化归档
3. 打开放文件的文件夹，把文件放进去
4. 预演维护
5. 正式维护

更短的入门说明见 `开始使用.md`。

## 前置条件

- Windows 系统（已安装 Python 3.9+）
- 日常使用：`点我开始.cmd`
- 高级命令行使用：`vault.py`

验证 Python 是否可用：
```
py --version
```

## 命令行快速上手

### 第一步：初始化

选一个文件夹作为归档根目录（例如 `D:\我的归档`），运行：

```
py vault.py init D:\我的归档
```

这会创建以下结构：
```
D:\我的归档\
├── master\          ← 把你的文件放这里
├── mirrors\
│   ├── mirror_1\    ← 系统自动维护的镜像 1
│   └── mirror_2\    ← 系统自动维护的镜像 2
├── snapshots\       ← 历史快照
└── .archive\        ← 系统文件（配置、日志、基准）
```

### 第二步：放入文件

把你要归档的文件放进 `master` 文件夹。支持子文件夹。

```
D:\我的归档\master\
├── 合同\
│   ├── 2024年采购合同.pdf
│   └── 2025年销售合同.pdf
├── 财务\
│   └── 年度报表.xlsx
└── 照片\
    └── 公司活动.jpg
```

### 第三步：运行维护

```
py vault.py maintain D:\我的归档
```

这一条命令会依次完成：

1. 核对 baseline 已跟踪的主副本文件
2. 原子创建并验证快照
3. 同步到两份镜像
4. 完整性校验
5. 必要时自动修复

最后输出结果：**通过** / **警告** / **需要处理**。

### 日常使用

每隔几个月运行一次 `maintain` 即可。新增文件会自动纳入；如果修改了 baseline 已跟踪的文件，先确认修改正确，再运行一次 `sync`，之后运行 `maintain`。GUI 中对应“同步已确认更改”按钮。

## 所有命令

| 命令 | 作用 | 会修改文件吗 |
|------|------|:---:|
| `init` | 初始化归档目录 | 是 |
| `maintain` | 通过主副本安全门后执行统一维护 | 是 |
| `sync` | 接纳并同步已确认的主副本更改 | 是 |
| `verify` | 只做校验，不修改任何东西 | 否 |
| `snapshot` | 只创建快照 | 是 |
| `repair` | 只做修复 | 是 |
| `status` | 查看当前状态 | 否 |

## 预演模式

`sync`、`snapshot`、`repair` 和 `maintain` 支持 `--dry-run`，只检查不实际操作：

```
py vault.py maintain D:\我的归档 --dry-run
```

**建议**：不确定时先加 `--dry-run` 看看会发生什么。

## 结果含义

- **通过**：一切正常
- **警告**：有需要注意的情况（如镜像中有多余文件），但不紧急
- **需要处理**：发现损坏或无法自动判定的冲突，需要你介入

## 删除保护

如果你从 master 中删除了已跟踪文件，`maintain` 会在创建快照、同步和更新 baseline 前停止，两份镜像仍保留原文件。

确认是有意删除后，先显式运行 `sync`；删除保护仍会保留镜像文件，后续 `maintain` 也不会自动清理这类警告项。再用 `repair --dry-run` 查看清理计划，确认后运行 `repair` 完成清理。

## 日志和报告

每次运行都会在 `.archive` 目录下生成：
- `logs/` — 详细日志（排查问题用）
- `reports/` — 摘要报告（看结果用）

## 常见问题

**Q: 我能直接修改 mirrors 或 snapshots 里的文件吗？**
不要。只修改 master。mirrors 和 snapshots 由系统维护。

**Q: 快照会占很多空间吗？**
默认最多保留 10 个完整快照。新快照只有在全部文件复制并校验成功后才会发布和触发旧快照淘汰。可在 `.archive/config.json` 中修改 `max_snapshots`。

**Q: 镜像有几份？能改吗？**
固定为两份：`mirror_1` 和 `mirror_2`。当前版本不再通过配置切换镜像数量。

**Q: 我想切换哈希算法，比如改成 blake2b？**
可以，把 `.archive/config.json` 里的 `hash_algorithm` 改成目标算法后，运行一次 `sync`，让系统按新算法重建最新基准。

**Q: 系统怎么判断哪个版本是对的？**
它不会再简单按“票数多就算对”。现在更接近“分层判定”：先看当前链（主副本 + 两份镜像）是否一致，再看基准和最新快照是否支持其中某个现存版本；如果历史证据不能明确支持，就停下来让你决定。
